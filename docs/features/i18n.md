# i18n — Internacionalización (ES / EN)

> **Toggle por usuario.** Cada profile tiene su propio idioma; afecta UI y generaciones de IA disparadas por ese usuario.

---

## 1. Decisiones arquitectónicas

| # | Decisión | Razón |
|---|----------|-------|
| 1 | **Cookie-based, sin `[locale]` en URL** | Arko es app autenticada; no hay SEO público. Evita refactor masivo de rutas. |
| 2 | **Locale per-user (no per-workspace)** | Modelo mental simple: "yo veo Arko en mi idioma". |
| 3 | **AI: forward-only** | Lo histórico se queda en el idioma en que se generó. Lo nuevo se genera en el idioma activo del usuario que dispara la acción. |
| 4 | **Pipelines automáticos (cron/sync)** | Toman el idioma del **owner** del workspace (`getWorkspaceLanguage`). |
| 5 | **Voice/tone parametrizado para Moka + specialists** | Prompts comunes con bloque de voz inyectable según locale. |
| 6 | **ADN onboarding: prompts paralelos ES/EN** | Conversation flow muy estructurado, traducción 1:1 más segura. |

---

## 2. Stack

- `next-intl` v4 (App Router, sin routing por URL)
- Tabla: `profiles.language` (`'es' | 'en'`, default `'es'`)
- Cookie: `NEXT_LOCALE` (no httpOnly, 1 año)

## 4. Flujo de resolución del idioma

### Login
1. Usuario se autentica → middleware lee `profiles.language` la primera vez (cookie ausente)
2. Setea cookie `NEXT_LOCALE` con ese valor (TTL 1 año)
3. `next-intl` lo recoge en `getRequestConfig`

### Toggle (Settings)
1. UI llama `updateUserLocale('en')`
2. Server action: UPDATE `profiles.language` + set cookie + `revalidatePath('/', 'layout')`
3. Próximo render usa el nuevo idioma

### Server components / API routes
- **UI server components** → `getLocale()` / `getTranslations()` de `next-intl/server` (lee cookie)
- **AI / generaciones** → `getUserLanguage()` de `@/i18n/server` (lee DB, autoritativo)
- **Cron / pipelines** → `getWorkspaceLanguage(workspaceId)` (toma idioma del owner)

---

## 5. Convenciones para extracción de strings

- Namespace por superficie: `auth.*`, `sidebar.*`, `dashboard.*`, `instagram.*`, etc.
- Strings cortos comunes en `common.*` (Save, Cancel, Loading, …)
- ICU plurals/select cuando aplique: `"items": "{count, plural, =0 {Sin items} one {# item} other {# items}}"`
- **NUNCA** concatenar strings traducidos. Usar interpolation: `t("welcome", { name })`

---

## 6. AI prompts

### Moka + specialists
- Prompts comunes en `src/services/arko-ai-prompts.ts` y `arko-ai-specialists.ts`
- Bloque `voice` parametrizado por locale (rioplatense para ES, neutral US English para EN)
- Lectura de idioma vía `getUserLanguage(userId)` antes de construir el prompt

### ADN onboarding (`adn-prompts.ts`)
- Dos archivos paralelos: `adn-prompts.es.ts` y `adn-prompts.en.ts`
- Selector exporta el correcto según `getUserLanguage()`

### Pipelines de análisis (Gemini/GPT)
- Se inyecta `Generate analysis in {{language}}` en cada prompt
- Idioma resuelto del usuario que dispara la acción, o del owner si es automático

## 7. Lo que NO está cubierto (por ahora)

- **Contenido IA histórico** sigue en el idioma original de generación
- **Hook classifier**: por ahora traduce todo a español; Fase 6 invierte la lógica para preservar original + traducir on-demand
- **Email templates / notifications**: aún no existen; cuando se implementen, usar `getUserLanguage()` del destinatario
