## Por qué F2.5

La lógica de llamadas a Meta Graph API está **duplicada y dispersa** en 5 callers vivos, en dos runtimes (Node/Next y Deno/edge). No hay backoff/retry en ningún lado, la versión `v25.0` está hardcodeada en ~7 sitios, y la paginación está copiada 4 veces. El objetivo: UN cliente Meta (versión, field-lists, auth, backoff, parser de errores, paginación) sin romper el sync de 6 clientes vivos.

## Callers vivos confirmados (evidencia archivo:línea)

1. **`supabase/functions/sync-instagram/index.ts`** (Deno) — el más pesado. `GRAPH_BASE` const (:28-29). ~20 call sites: `/media`, `/children`, `/insights` (reels+posts+account+followers+demographics), `/stories`, `/ads`, ads `/insights`. Paginación `while(url)` duplicada en :1592/:1714/:1738/:1755. Parser token-expired `isTokenExpiredError` (:1808-1816). **Sin backoff/retry.**
2. **`supabase/functions/refresh-meta-tokens/index.ts`** (Deno) — `/oauth/access_token?fb_exchange_token` (:73, versión hardcodeada inline). Único con timeout (15s, :81). **🐞 BUG: marca `status='expired'` ante CUALQUIER fallo (:89-104)**, incluido 5xx/rate-limit transitorio → falsos-desconectados para los 6 clientes.
3. **`src/app/api/v1/auth/meta/callback/route.ts`** (Node) — flujo OAuth de alta: 7 fetch inline, `v25.0` hardcodeada 6 veces (:71,89,102,112,118,129,146). Duplica el endpoint `/oauth/access_token?fb_exchange_token` con refresh. Solo valida error en el primer exchange (:80).
4. **`src/app/api/v1/meta/explorer/route.ts`** (Node) — passthrough de debug. `GRAPH_BASE` const copiada (:14-15). No escribe a DB, invocación manual. **El caller más seguro para migrar primero.**
5. **`src/app/api/v1/auth/meta/connect/route.ts`** (Node) — NO llama Graph; arma la URL de `www.facebook.com/v25.0/dialog/oauth` (:84). Solo unificar el literal de versión.

## Diseño del cliente

**DOS artefactos espejo con contrato idéntico** (NO symlink, NO import cruzado — la pared de runtimes lo prohíbe: `tsconfig.json:33` excluye `supabase/functions`; `_shared` usa `Deno.env` + imports esm.sh + extensión `.ts`):
- **Node:** `src/lib/meta/` → `constants.ts` + `client.ts` (alias `@/lib/meta`).
- **Deno:** `supabase/functions/_shared/meta/` → `constants.ts` + `client.ts` (import relativo, estilo Deno, como ya hace con `types.ts`).

**`constants.ts` (puro, sin runtime — perfil idéntico a `_shared/types.ts`):**
- `META_GRAPH_VERSION = 'v25.0'`, `GRAPH_BASE`.
- Field-lists hoy duplicadas dentro de sync-instagram: `MEDIA_FIELDS` (:1588), `CREATIVE_FIELDS` (:1710/:1725), `AD_INSIGHT_FIELDS` (:1735), `AD_DAILY_FIELDS` (:1752). **Copiar TEXTUALMENTE.**
- Tabla de error-codes: `190/102/104 → needs_reauth`, `4/17/32/613/80004 → rate_limit`, `100/3001 → unsupported_metric`.

**`client.ts`:**
- `MetaApiError extends Error` con `kind: 'rate_limit'|'needs_reauth'|'server'|'client'`, `code`, `status`, `retryable`, `raw`.
- `metaFetch<T>(path, { token, params?, version?, method?, signal?, maxRetries? })`: token como query param (patrón actual, NO cambiar a Bearer en F2.5), URL con `GRAPH_BASE`, lee `data.error` una vez, normaliza con `classifyMetaError()`, backoff exponencial con jitter SOLO si `retryable` (molde `gemini-video.service.ts:425-447`, base ~1.5s, factor 2, **budget corto <~6.5s / 2 reintentos** por el timeout del edge ~120-150s, respeta `Retry-After`), `AbortSignal.timeout` por intento.
- `metaFetchPaged<T>(path, opts)`: encapsula el `while(url){...; url=data.paging?.next}`.
- `classifyMetaError(graphError, httpStatus)`: única fuente de verdad.

**NO centralizar** (específico de cada caller, exponer como params): cadenas de fallback de insights (:1602-1698), flujo OAuth del callback, `resolveAdToReel`/`getActionMetricValue`/`normalizeInstagramUrl`, el modo proxy del explorer. **NO tocar** el comentario `v22.0` (:1202, es doc de stories).

**Control de drift entre las 2 copias:** test de paridad Vitest (CI) que lee ambos `constants.ts` como texto y hace assert de igualdad de valores clave (versión + field-lists + error-codes); falla el build si divergen. Sustituye el symlink prohibido.

Crear `src/lib/meta/constants.ts` (Node) con `META_GRAPH_VERSION`, `GRAPH_BASE`, field-lists copiadas TEXTUALMENTE de sync-instagram, y la tabla de error-codes. ## Riesgos clave
- **Drift entre las 2 copias** → mitigado por test de paridad en CI.
- **Sin PITR:** reversión del edge = redeploy del `index.ts` anterior (documentar el SHA estable antes de cada cut-over).
- **Backoff mal acotado** puede timeoutear el sync (~120-150s) → budget corto y explícito.
- **El backoff cambia comportamiento** (reintenta donde antes fallaba) → verificar paridad de SHAPE, no solo presencia.
- **Redeploy durante cron** (0/6/12/18 UTC) puede pisar una corrida → cut-over fuera de esas horas.
- **`callback` no es retryable** (OAuth interactivo) → un bug bloquea onboarding; validar end-to-end en Dev.
- **Over-engineering:** NO centralizar fallbacks de insights / OAuth / matching de ads (acoplaría callers).

## 🐞 Bonus descubierto durante el recon
`refresh-meta-tokens/index.ts:89-104` marca `status='expired'` ante cualquier fallo del exchange (incluido 5xx/rate-limit transitorio). Esto puede estar **desconectando clientes sin razón real**. Se corrige en F2.5-4 (solo marcar expired ante `needs_reauth`/190/OAuthException; dejar que el cron reintente ante transitorios). **Si se quiere, este fix se puede hacer aislado antes que todo F2.5** — es acotado y de alto valor.
